# Race Rewind вЂ” Architecture Overview

A time-travel F1 stats site: pick any historic race and see standings, driver/team stats, and championship context _as they were at that point in time_. Built for rewatching old seasons without spoilers.

## Goals & Constraints

- **Cheap to run**: target $0/month at low-to-moderate traffic; never pay for compute on idle pages
- **Fast everywhere**: ad-supported, so Core Web Vitals matter for both UX and RPM
- **SEO-friendly**: static HTML where possible, clean URLs, proper metadata, structured data
- **Time-travel correct**: every stat is computable "as of race N" вЂ” no denormalized current-state totals
- **Legal-safe**: no logos, no F1/team trademarks beyond nominative use, no copyrighted photos, attribution where required
- **Simple to maintain**: one artifact (the SQLite), one shot to re-ingest, no infrastructure to manage

## Tech Stack

| Layer     | Choice                                           | Why                                                                             |
| --------- | ------------------------------------------------ | ------------------------------------------------------------------------------- |
| Framework | Astro (`output: 'static'`)                       | Islands architecture, near-zero JS, great SEO defaults, full static output      |
| Hosting   | Cloudflare Pages                                 | Unmetered bandwidth, free tier covers the whole stack                           |
| Database  | SQLite (committed to repo)                       | Fully derived from source CSVs; ~10вЂ“15 MB; no infrastructure, no sync, no drift |
| DB driver | `@libsql/client` + Drizzle ORM                   | Reads the committed .sqlite during `astro build`; never shipped to the browser  |
| Ingest    | TypeScript + Drizzle ORM (`tsx`, `csv-parse`)    | Shares the same schema definitions as the Astro app; single source of truth     |
| Analytics | Cloudflare Web Analytics + Google Search Console | Privacy-friendly, no cookie banner needed, SEO insights                         |
| Ads       | Google AdSense в†’ Ezoic в†’ Mediavine (at scale)    | Standard progression as traffic grows                                           |
| Domain    | Cloudflare Registrar                             | At-cost pricing, free WHOIS privacy                                             |

**Not used (and why):** Cloudflare D1, Cloudflare Workers, Wrangler. The data is fully historical and immutable, so all ~28k pages can be pre-rendered at build. SSR and its associated infrastructure add complexity without providing value for this use case.

## Data Model

### Timeline axis

The entire site is organised around a single global race counter вЂ” `race_number` вЂ” sourced verbatim from the Jolpica F1 database (`formula_one_round.race_number`, 1..1171 across 1950вЂ“2026). "Stats as of race X" means `WHERE race_number <= X`. No date parsing, no timezone headaches.

Cancelled rounds are excluded from the timeline (they have a null `race_number` in the Jolpica dump).

### Identifier strategy

Every entity carries three identifiers:

| Identifier              | Purpose                                     | Example                                           |
| ----------------------- | ------------------------------------------- | ------------------------------------------------- |
| `id` (integer PK)       | Stable join key inside the DB               | `1234`                                            |
| `jolpica_id` (integer)  | Back-reference to source CSV `id` column    | `847`                                             |
| `jolpica_api_id` (text) | Back-reference to Jolpica HTTP API `api_id` | `driver_NNme7ruE`                                 |
| `slug` (text)           | URL segment for our site                    | `michael-schumacher`, `1998-hungarian-grand-prix` |

Slugs are generated deterministically at ingest and stable across re-runs. For races: `{year}-{slugified-name}` (e.g. `1998-hungarian-grand-prix`). For drivers: `{forename}-{surname}` lowercased. For teams/circuits: prefer Jolpica's `reference` column.

### Per-lap / pit-stop data

Not stored as tables in v1 вЂ” race-level aggregates cover the use cases:

- **Gap to leader**: `race_results.time` (already in session entries, formatted `+12.345s`)
- **Total laps**: `race_results.laps_completed`
- **Pit stops**: `race_results.pit_stop_count` (aggregated at ingest from the pitstop CSV)
- **Fastest lap**: `race_results.fastest_lap_rank = 1`

The raw CSVs remain in `ingest/jolpica-dump/` for future lap-chart features.

### Derived columns and tables (computed at ingest)

```sql
-- races.prev_race_in_season: foreign key to the previous race in the same season
-- (NULL for round 1 of each season); enables O(1) "standings going into this race" lookups

-- races.is_final_round: 1 for the last race of each season; used for WDC count queries

-- driver_career_progression: one row per (driver_id, race_number) the driver participated in
-- Cumulative stats including that race: cum_starts, cum_wins, cum_podiums, cum_poles,
-- cum_fastest_laps, cum_points, cum_championships
-- Career stats "going into race R" = latest row WHERE race_number < R
```

```sql
-- Championship standings at race R
SELECT * FROM driver_standings WHERE race_number = ?

-- "Which team did driver D drive for at race R?"
SELECT team_id FROM round_entries WHERE driver_id = ? AND race_number = ?

-- WDC count for driver D (how many seasons they ended P1)
SELECT COUNT(*) FROM driver_standings ds
  JOIN races r ON r.race_number = ds.race_number
  WHERE ds.driver_id = ? AND ds.position = 1 AND r.is_final_round = 1
```

## Rendering Strategy

**Fully static.** All pages are pre-rendered at `astro build` from the committed SQLite. No SSR, no Workers, no D1.

### Pre-rendered pages (~28k total)

| Route                          | Count  | Notes                                                 |
| ------------------------------ | ------ | ----------------------------------------------------- |
| `/`                            | 1      | Home                                                  |
| `/seasons/`                    | 1      | Season index                                          |
| `/seasons/[year]/`             | ~77    | Season overview                                       |
| `/seasons/[year]/[race-slug]/` | ~1,171 | Race-weekend page                                     |
| `/drivers/`                    | 1      | Driver index                                          |
| `/drivers/[slug]/`             | ~879   | Driver career view                                    |
| `/drivers/[slug]/[race-slug]/` | ~26k   | Time-travel view (driver Г— race they participated in) |
| `/teams/[slug]/`               | ~215   | Team overview                                         |

**Build time estimate:** ~2вЂ“5 min on Cloudflare Pages' standard builder (4 vCPU, 4 GB RAM). Well within the 20-min timeout and 500 builds/month free quota.

**Cache behaviour:** Cloudflare Pages serves all static assets from the edge. Since historic data is immutable, pages cached by CDN are valid until the next deploy. No `Cache-Control` tuning needed for static assets вЂ” Pages sets optimal headers by default.

### Race-weekend page (the centrepiece)

`/seasons/[year]/[race-slug]/` renders the full weekend in chronological order:

1. **Qualifying** вЂ” `qualifying_results` ordered by position; Q1/Q2/Q3 times shown where available
2. **Sprint qualifying** вЂ” `sprint_qualifying_results` (only if `race.has_sprint`)
3. **Sprint race** вЂ” `sprint_results` (only if `race.has_sprint`)
4. **Race** вЂ” `race_results` ordered by position; includes gap, laps, status, pit stops, fastest lap
5. **Standings impact** вЂ” diff between `driver_standings` at `race_number - 1` (entering the round) and `race_number` (after the round)
6. **Championship tables** вЂ” full driver and constructor standings as of this race

All six queries are keyed lookups on indexed columns; no full-table scans.

## Data Ingestion Pipeline

The pipeline converts the committed CSV dump into `data/race-rewind.sqlite`. It is the only source of DB content вЂ” the SQLite is fully derived and can be reproduced at any time.

```
ingest/jolpica-dump/YYYY-MM-DD/*.csv
        в†“
pnpm ingest  (tsx ingest/build_db.ts)
        в†“
data/race-rewind.sqlite  (committed to repo, ~17 MB)
        в†“
astro build  (reads SQLite via @libsql/client + Drizzle ORM at build time)
        в†“
dist/  (fully static site, ~28k HTML files)
```

**Pipeline phases:**

1. Drop existing SQLite; create fresh client; apply PRAGMAs (`journal_mode=OFF`, `synchronous=OFF`, `foreign_keys=ON`)
2. Apply Drizzle migration (`drizzle/0000_right_catseye.sql`) вЂ” schema is the single source of truth in `src/db/schema.ts`
3. Load reference tables (`championship_systems`, `point_systems`)
4. Load dimensions (`seasons`, `circuits`, `drivers`, `teams`) вЂ” generate slugs, build `jolpica_id в†’ internal_id` maps
5. Load timeline (`races`, `sessions`) вЂ” filter out cancelled rounds
6. Load entries (`round_entries`, `session_entries`) вЂ” resolve mid-season team changes
7. Build denormalised result tables (`race_results`, `sprint_results`, `qualifying_results`, `sprint_qualifying_results`) вЂ” pivot Q-sessions for modern era
8. Load standings snapshots (`driver_standings`, `team_standings`) вЂ” dedup to post-final-session per round
9. Build derived columns (`races.prev_race_in_season`, `races.is_final_round`) and `driver_career_progression` table via window functions
10. `PRAGMA foreign_key_check` в†’ `VACUUM` в†’ `ANALYZE`

**Re-ingest contract:** drop-and-rebuild every time. To update for a new race: add the new Jolpica dump folder and re-run. No incremental logic, no migrations.

**Schedule:**

- Full historical import (1950вЂ“present): one-time, already done via committed CSV dump
- GitHub Actions weekly during F1 season вЂ” fetch latest Jolpica dump, re-run ingest, commit updated SQLite, trigger Pages redeploy
- Manual runs for backfills or corrections

### Adding per-lap data later

When lap-chart pages are wanted: add `laps` and `pit_stops` tables to `src/db/schema.ts`, run `pnpm drizzle-kit generate` to produce a new migration, add a `laps.ts` transform module under `ingest/transform/`, and re-run `pnpm ingest`. The CSV data is already in `ingest/jolpica-dump/`. Expected size increase: +35вЂ“60 MB.

## SEO Foundation

### Per-Page Requirements

Every page renders with:

- Unique `<title>` (e.g. "1998 Hungarian Grand Prix вЂ” Qualifying, Results & Standings")
- Unique meta description
- Canonical URL
- Open Graph tags (og:title, og:description, og:image)
- JSON-LD structured data:
  - `SportsEvent` for races
  - `Person` for drivers
  - `SportsTeam` for constructors
  - `BreadcrumbList` site-wide

### Site-Wide

- `sitemap.xml` via `@astrojs/sitemap`
- `robots.txt` allowing all crawlers, pointing to sitemap
- Submitted to Google Search Console + Bing Webmaster Tools
- Dense internal linking (race в†” drivers в†” team в†” season в†” adjacent races)

### Content Layer (Manual)

Pillar articles in `src/content/` (Astro content collections, MDX) вЂ” pre-rendered at build alongside the data pages:

- "How to rewatch [season] F1 spoiler-free"
- "F1 mid-season comebacks"
- "Every notable mid-season team change"

Articles reference races/drivers by slug (e.g. `/seasons/1994/san-marino-grand-prix/`) so links stay valid across ingest re-runs.

## Deployment

1. Push to `main` в†’ Cloudflare Pages CI runs `pnpm build` (which includes the SQLite read via `better-sqlite3`) в†’ deploys static output
2. `data/race-rewind.sqlite` is committed to the repo, so the Pages build has it available without any extra setup
3. New race data: GitHub Actions runs `pnpm ingest`, commits updated SQLite, and pushes вЂ” Pages auto-deploys
